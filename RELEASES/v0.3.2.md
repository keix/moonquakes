# Moonquakes v0.3.2 — Structure Before Speed

Moonquakes continues the post-freeze cleanup of the v0.3 line.

This release is a maintenance patch on top of v0.3.1.
It does not widen the Lua-visible compatibility surface.
It continues the internal reorganization needed before `v0.4.x` performance work begins.

## Scope of v0.3.2

v0.3.2 extends the v0.3 line by making execution, builtin dispatch, and error handling boundaries easier to follow:

- Grouped builtin dispatcher entrypoints into a more uniform public layer
- Simplified builtin library registration in `src/builtin/dispatch.zig`
- Consolidated shared Lua-exception classification across main execution, reentrant calls, and coroutine resume
- Extracted protected-call bootstrap logic into a focused VM module
- Continued reshaping `src/vm/mnemonics.zig` around `do(...)`, `opMNEMONIC`, and shared helpers
- Added execution-facing comments that describe stack effects and semantic boundaries at opcode level
- Updated internal scope and error-semantics notes to match the current structure

This version focuses on internal readability, refactor safety, and preparing the codebase for the optimization line.

## Compatibility

Lua 5.4 Compatibility Contract

- The Lua-visible compatibility contract frozen in `v0.3.0` remains the boundary
- `_VERSION == "Lua 5.4"` remains unchanged
- The aggregate suite in `passing/all.lua` remains passing
- `zig build test` passes the full suite
- No new language features or public runtime surface are introduced in this patch

This release is intentionally structural.
Its purpose is to make the implementation easier to evolve without changing validated behavior.

## Internal Cleanup

### Builtin surface and registration

- Added consistent file-level structure to builtin modules so shared helpers and dispatcher entrypoints are easier to distinguish
- Grouped dispatcher-facing `native*` entrypoints together across the builtin set
- Reworked `src/builtin/dispatch.zig` so library registration is more table-driven and special initialization remains explicit

### Error semantics and protected calls

- Consolidated shared runtime-error classification helpers in the VM
- Introduced a shared Lua-exception disposition step while keeping caller-specific control-flow mapping local
- Brought `mnemonics.zig`, `call.zig`, and coroutine resume paths onto the same semantic vocabulary
- Moved protected-call bootstrap helpers into `src/vm/protected_call.zig`

### Execution-structure cleanup

- Continued separating `src/vm/mnemonics.zig` into shared helper sections, `do(...)`, and opcode handlers
- Standardized opcode handler naming around `opMNEMONIC`
- Added opcode-level comments for stack behavior and semantic role
- Reduced the amount of non-dispatch logic directly embedded in the top-level execution flow

### Documentation and scope alignment

- Updated `errors.md` to reflect the current shared semantic-core approach
- Updated `docs/moonquakes-v0.3x-scope.md` so the documented cleanup strategy matches the implementation
- Clarified that `v0.3.x` remains the structural cleanup line, while throughput work belongs to `v0.4.x`

## Why this release matters

`v0.3.0` froze behavior.
`v0.3.1` hardened internal ownership boundaries.
`v0.3.2` makes the remaining execution and error paths easier to navigate and safer to change.

The immediate benefit is still not new Lua functionality.
The benefit is that future optimization work can start from clearer opcode boundaries, clearer builtin registration structure, and clearer shared error semantics.

## Differences from Lua 5.4

Known Differences

| Behavior                    | Lua 5.4   | Moonquakes v0.3.2                     |
| --------------------------- | --------- | ------------------------------------- |
| Weak tables (`__mode`)      | Supported | Not implemented                       |
| Debug metadata completeness | Full      | Partial compatibility                 |
| Traceback formatting        | Full      | Semantic shape tested, not byte-exact |
| `package` edge compatibility| Full      | Core behavior covered, details remain |
| Public C API                | Available | Not yet implemented                   |

## What v0.3.2 Is Not

The following remain outside the scope of this release:

- throughput optimization
- call/return fast paths
- instruction-dispatch experiments
- hot-path table/metamethod specialization
- ABI stability
- public C API finalization
- weak table semantics (`__mode`)
- complete debug metadata compatibility

## Design Position

v0.3.2 marks the point at which:

- builtin organization is more uniform across the standard library surface
- shared error semantics are explicit across main execution, reentrant call, and coroutine paths
- protected-call bootstrap logic is no longer buried in the main opcode file
- `mnemonics.zig` is much closer to a `do + opMNEMONIC + shared helper` layout

This release does not broaden the contract.
It reduces structural friction ahead of `v0.4.x`.
