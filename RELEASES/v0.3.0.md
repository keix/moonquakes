# Moonquakes v0.3.0 — First Quarter (Frozen)

Moonquakes enters its first compatibility freeze.

This release does not primarily expand the language surface.
It defines and freezes the Lua-visible contract that future optimization work
must preserve.

## Scope of v0.3.0

v0.3.0 extends the validated runtime from "passing now" to "staying passing":

- Introduced `src/tests/api` as a black-box compatibility layer for builtin and core runtime behavior
- Declared the Lua-visible compatibility contract frozen as of `v0.3`
- Kept the official Lua 5.4 aggregate suite passing through `passing/all.lua`
- Expanded regression coverage for metatables, numeric behavior, iteration, protected calls, loading, and GC control
- Tightened several user-visible edge cases discovered while building the API suite

This version focuses on freezing behavior before optimization.

## Compatibility

Lua 5.4 Compatibility Contract

- Moonquakes presents itself as Lua 5.4 from Lua code
- `_VERSION == "Lua 5.4"`
- The official aggregate suite in `passing/` passes end-to-end
- `src/tests/api` now defines the frozen black-box contract for the builtin/runtime surface
- `zig build test` currently passes `357/357` tests

The compatibility boundary is now explicit.
Future internal changes are expected to preserve behavior covered by the upstream suite and `src/tests/api`.

## API Freeze

### Frozen surface

The new API contract layer covers:

- `global`
- `modules`
- `debug`
- `coroutine`
- `string`
- `table`
- `math`
- `utf8`
- `io`
- `os`
- `metatable`
- `numeric`
- `iteration`
- `error`
- `gc`

Taken together, these tests freeze:

- builtin library semantics
- result arity and return ordering
- metatable dispatch and raw-access boundaries
- numeric conversion and equality behavior
- `next` / `pairs` / `ipairs` iteration contracts
- protected-call and traceback shape
- module loading and loader cache behavior
- host-facing `io`, `os`, and GC control surfaces

### Why this release matters

`v0.4.0` is expected to focus on optimization.
`v0.3.0` is the release that defines what those optimizations are not allowed to change.

This is a behavior freeze, not an implementation freeze.
The VM, compiler, allocator, and runtime internals may still evolve substantially.

## Behavioral Hardening

The API-freeze push also exposed and fixed several observable issues:

- Corrected `ipairs` termination so finished iteration clears trailing results correctly
- Hardened bytecode dump/load ownership so serialized chunks do not corrupt proto-owned buffers
- Tightened `debug` local/userdata behavior through direct black-box tests
- Locked down CLI compatibility around official stand-alone interpreter expectations

These fixes were made to stabilize the contract that the new suite now protects.

## Differences from Lua 5.4

Known Differences

| Behavior                         | Lua 5.4   | Moonquakes v0.3.0                    |
| -------------------------------- | --------- | ------------------------------------ |
| Weak tables (`__mode`)           | Supported | Not implemented                      |
| Debug metadata completeness      | Full      | Partial compatibility                |
| Traceback formatting             | Full      | Semantic shape tested, not byte-exact |
| `package` edge compatibility     | Full      | Core behavior covered, details remain |
| Public C API                     | Available | Not yet implemented                  |

## What v0.3.0 Is Not

The following remain outside the scope of this release:

- ABI stability
- Public C API finalization
- Weak table semantics (`__mode`)
- Complete debug metadata compatibility
- Full `package` compatibility beyond currently tested behavior
- Every platform-specific `io` / `os` edge case

## Design Position

v0.3.0 marks the point at which:

- Moonquakes has a frozen Lua-visible compatibility contract
- Upstream-style aggregate validation and API-level contract testing reinforce each other
- Optimization work can proceed against an explicit behavioral boundary

This release is the transition from compatibility effort to compatibility discipline.
