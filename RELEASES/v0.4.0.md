# Moonquakes v0.4.0 — Waxing Gibbous

Moonquakes enters the first `v0.4.x` release line.

This release keeps the Lua-visible compatibility contract established in `v0.3.0`, while advancing two internal goals:

- make the VM execution core easier to optimize safely
- bring the garbage collector to a documented, test-backed release baseline

`v0.4.0` is not a language-expansion release. 

It is the point where the runtime becomes more explicit about execution ABI, GC invariants, and optimization boundaries.

## Scope of v0.4.0

`v0.4.0` extends the post-freeze runtime by making hot execution paths and GC semantics more explicit and more stable:

- Consolidated suspended opcode state into a first-class `Continuation` model
- Split call-frame state from continuation state across VM execution
- Unified return-path preparation and caller result transport
- Consolidated call setup for Lua closures, native closures, tail calls, and protected calls
- Reduced duplication across main execution, coroutine resume, and reentrant call loops
- Introduced shared synthetic-frame helpers for coroutine and protected-call bootstrap
- Normalized metamethod and `__call` dispatch setup around shared callable-resolution helpers
- Enforced a single GC mutation boundary: all GC-visible writes now go through `gc` mutation helpers
- Implemented incremental GC stepping and generational cycle handling
- Added weak-table, finalizer, remembered-set, and barrier-invariant tests
- Documented GC architecture and the `v0.4.0` GC freeze boundary

This release focuses on semantic clarity, test coverage, and safe optimization
headroom.

## Compatibility

Lua 5.4 Compatibility Contract

- The Lua-visible compatibility contract frozen in `v0.3.0` remains the boundary
- `_VERSION == "Lua 5.4"` remains unchanged
- The aggregate suite in `passing/all.lua` remains passing
- `zig build test` passes `384/384` tests
- No new Lua-visible language surface is introduced in this release

This release expands internal guarantees, not public language scope.

## VM Execution Baseline

### Continuation and frame boundaries

- Replaced scattered `pending_*` state with a unified `Continuation` model
- Kept `CallInfo` focused on stable frame ownership while suspended instruction
  state lives separately
- Reduced opcode-specific resume logic duplication for return, compare, and concat paths

### Call and return ABI cleanup

- Unified `RETURN`, `RETURN0`, and `RETURN1` setup and completion paths
- Consolidated call-frame staging for Lua and native calls in `src/vm/call.zig`
- Shared more execution-loop structure across main execution, coroutine resume,
  and reentrant calls
- Simplified synthetic `CALL/RETURN` bootstrap frames for coroutine and protected-call flows

### Metamethod and callable dispatch cleanup

- Separated `__call` target resolution from invocation
- Unified callable classification for Lua closures and native closures
- Reduced duplicated stack-setup logic across metamethod call paths
- Normalized `__index` / `__newindex` dispatch and `ipairs` fallback behavior

These changes do not freeze the execution ABI as public API, but they define a clearer internal baseline for optimization work in the rest of the `v0.4.x` line.

## Garbage Collector Baseline

### What landed in `v0.4.0`

- Added incremental GC step progression with explicit `idle -> mark -> sweep -> idle` state transitions
- Added generational mode with minor and major cycles
- Added centralized GC mutation helpers in `src/runtime/gc/mutation.zig`
- Routed production GC-visible writes through the GC mutation boundary
- Added remembered-set handling for old tables that reference younger objects
- Added weak-table cleanup and finalizer-queue tests for generational behavior
- Added barrier tests for table, upvalue, userdata, file, and thread-related writes
- Added GC architecture documentation and an explicit GC freeze-boundary document

### GC release position

For `v0.4.0`, the GC should be treated as:

- semantically stable at the Lua-visible layer
- structurally stable at the VM/runtime contract layer
- still open to internal optimization and policy refinement

What is now treated as release-grade:

- `collectgarbage` API shape and observable control behavior
- weak-table semantics
- queued-finalizer semantics
- explicit root-provider and VM-root boundaries
- tri-color and remembered-set correctness invariants
- centralized GC-visible mutation rules

What remains open after `v0.4.0`:

- non-table old-container policy in minor cycles
- remembered-set implementation details
- step-budget tuning and scheduling heuristics
- collector performance optimization

## Documentation

`v0.4.0` adds or updates internal design documentation for:

- VM refactor status and execution cleanup
- GC architecture
- GC freeze boundary
- `v0.4.x` optimization scope

This is intentional.

The release aims to make future optimization and correctness work operate
against written boundaries rather than implicit assumptions.

## Why this release matters

`v0.3.0` froze Lua-visible behavior.
`v0.3.x` reduced internal structural friction.
`v0.4.0` establishes the first release-grade execution and GC baseline that
future optimization work can build on.

The immediate benefit is not a new language feature.
The benefit is that call/return paths, coroutine/protected-call boundaries, and
GC semantics are now explicit enough to optimize without guessing where the
compatibility line is.

## Differences from Lua 5.4

Known Differences

| Behavior                         | Lua 5.4   | Moonquakes v0.4.0                    |
| -------------------------------- | --------- | ------------------------------------ |
| Debug metadata completeness      | Full      | Partial compatibility                |
| Traceback formatting             | Full      | Semantic shape tested, not byte-exact |
| `package` edge compatibility     | Full      | Core behavior covered, details remain |
| Public C API                     | Available | Not yet implemented                  |

Weak tables and finalizer semantics are now implemented and validated as part
of the `v0.4.0` baseline.

## What v0.4.0 Is Not

The following remain outside the scope of this release:

- public C API stabilization
- final ABI stabilization for embedders
- broad parser or compiler redesign
- VM restructuring whose primary goal is readability rather than performance or correctness
- collector policy finalization beyond the documented freeze boundary
- every platform-specific `io` / `os` edge case

## Design Position

`v0.4.0` marks the point at which:

- the frozen Lua-visible compatibility line remains intact
- the VM execution core is organized around clearer continuation, call, and return boundaries
- the garbage collector has documented invariants, mutation boundaries, and release-grade tests
- future optimization work can proceed against explicit runtime and GC contracts

This release does not widen the language contract.
It makes the implementation disciplined enough to optimize without losing it.
GC is no longer an implementation detail. It is now an explicit contract.
